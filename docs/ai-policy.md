# Axmol Engine AI Contribution Policy

To maintain the high quality, performance, and stability of the Axmol Engine, we welcome all meaningful contributions from the community. With the rapid adoption of generative AI tools, we want to **proactively** establish clear boundaries to ensure AI assistance enhances—rather than degrades—the project's development workflow.

Managing a complex, cross-platform C++17/20 game engine requires significant maintainer time and code-review bandwidth. To prevent future resource drain and avoid notification spam for project watchers, we enforce the following policy regarding AI-assisted and AI-generated content.

---

## 1. Core Philosophy
* **AI is a Tool, Not a Contributor:** AI can assist *you*, but *you* (the human) are entirely responsible for the code you submit. 
* **Human Accountability:** Every line of code submitted must be thoroughly understood, manually audited, and locally verified by the author.
* **Respect Maintainer Bandwidth:** "Drive-by" PRs generated with a single prompt to "fix bugs" or perform sweeping "refactorings" without a deep understanding of the engine's architecture will be rejected.

---

## 2. Acceptable vs. Unacceptable Use

### Acceptable Use ✅
* Using AI assistants (Cursor, Copilot, Claude, Gemini, etc.) inside your local IDE to speed up typing, generate boilerplate, or explain existing engine patterns.
* Aligning your local AI workspace context using the repository's `docs/ai-instructions.md` to ensure it targets modern `ax::` namespaces and C++17/20 paradigms.
* Manually reviewing, modifying, and locally compiling every line of AI-suggested code before committing.

### Unacceptable Use ❌
* **Raw Copy-Pasting:** Submitting AI-generated code snippets directly without manual verification, proper testing, or local compilation.
* **Automated/Drive-By PRs:** Submitting PRs generated by automated scraping tools, external bots, or mass-PR generation scripts.
* **Hallucinated Refactoring:** Submitting PRs that "fix" non-existent issues, introduce hallucinated APIs, or use legacy Cocos2d-x structures (`cocos2d::`).
* **Unverifiable Submissions:** Submitting complex changes to core subsystems (RHI, platform-specific glue, memory management) where the author cannot explain *why* the change works or how it impacts performance.

---

## 3. How We Identify Low-Quality AI PRs
Maintainers look for specific red flags that indicate an unverified, AI-dumped submission:
1. **Hallucinated APIs or Comments:** Use of methods, macros, or properties that do not exist in Axmol (often falling back to old Cocos2d-x names or entirely fictional structures).
2. **Generic/AI-Style Descriptions:** PR descriptions using overly formal, repetitive, or generic summaries (e.g., *"This PR enhances modularity and robustness by optimizing critical variables..."*).
3. **Syntax or Logical Flaws:** Code that fundamentally breaks C++ memory models, fails to compile via CMake, or introduces obvious race conditions.
4. **Inability to Explain Changes:** When asked for clarification on review comments, the author gives vague, generic answers or is unable to provide technical depth.

---

## 4. Policy Enforcement & Consequences
To maintain a clean repository and protect contributor resources, the following actions will be taken for any PR suspected of violating this policy:

* **Immediate Closure:** Any PR identified as a raw, low-quality, or automated AI submission will be **closed immediately without a formal code review**.
* **No Argument Triage:** Maintainers will mark the PR as `spam` or `invalid` and close the thread. We will not spend time debug-proving why an AI-generated snippet is incorrect if the author hasn't done the basic work of compiling and testing it first.

---

## 5. Submission Checklist for Contributors
To ensure your PR is well-received, please confirm the following before submitting:
- [ ] **Local Verification:** I have compiled the engine cleanly with CMake using this patch.
- [ ] **Platform Testing:** I have verified this change on at least one target platform (e.g., Windows, Android, iOS, macOS, Linux).
- [ ] **Ownership:** I fully understand every line of code submitted and am prepared to maintain or modify it during review.
- [ ] **No Legacy Code:** I have checked that no legacy Cocos2d-x namespaces (`cocos2d::`) or deprecated macros have been introduced.

---

Thank you for respecting the Axmol community, our maintainers' time, and the engineering standards of this project!
